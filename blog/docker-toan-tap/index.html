1. Vấn Đề Trước Docker — "It Works on My Machine"

Bạn đã từng gặp cảnh này chưa?

Dev A (macOS): "Mình chạy được rồi, không có bug gì!"
Dev B (Ubuntu): "Sao mình chạy lại lỗi nhỉ?"
Server Production (CentOS): "Application crash khi deploy!"
QA (Windows 10): "Mình cài xong mà không lên được..."

Vấn đề không phải ở code — mà ở môi trường. Mỗi máy có phiên bản OS, runtime, thư viện hệ thống và biến môi trường khác nhau. Đây là dependency hell mà developer nào cũng từng trải qua.

Dành cho: Developer muốn hiểu tại sao container nhẹ và cách ly được mà không cần VM.

Container Linux được xây dựng trên 3 tính năng cốt lõi của Linux kernel:

1. Linux Namespaces — Tạo sự cô lập

Namespace cho phép kernel tạo ra các "góc nhìn" riêng biệt về tài nguyên hệ thống cho mỗi process:

PID
Process IDs — cây process riêng, PID 1 = main process
NET
Network — interface riêng (eth0, lo), IP riêng
MNT
Mount points — filesystem riêng
UTS
Hostname — hostname riêng
IPC
Inter-process communication — message queue riêng
USER
User/Group IDs — map UID container sang UID host

2. cgroups — Giới hạn tài nguyên

cgroups cho phép giới hạn và theo dõi CPU, RAM của một nhóm process. Không có cgroups, một container bị bug vô hạn loop có thể ăn hết RAM của cả server production!

bash
# Docker tạo cgroup limits khi bạn chạy:
docker run --memory="512m" --cpus="1.5" nginx
# /sys/fs/cgroup/memory/docker//memory.limit_in_bytes = 536870912
# /sys/fs/cgroup/cpu/docker//cpu.cfs_quota_us = 150000

3. Union File System — Image theo layers

Docker dùng overlay filesystem để tạo image từ nhiều layers — nếu 10 container cùng dùng base image, Docker chỉ lưu một lần trên disk:

FROM eclipse-temurin:21-jdk-alpine Base · Read-only · Shared
RUN ./mvnw dependency:go-offline Layer 1 · Read-only
COPY src ./src && RUN mvnw package Layer 2 · Read-only
Container Writable Layer R/W · Per-container · Ephemeral

3. Cài Đặt Docker

bash — Ubuntu/Debian
# Bước 1: Gỡ phiên bản cũ
sudo apt remove docker docker-engine docker.io containerd runc

# Bước 2-4: Thêm Docker repository
sudo apt update && sudo apt install -y ca-certificates curl gnupg
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Bước 5: Cài Docker Engine
sudo apt update && sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Bước 6: Cho phép user chạy Docker không cần sudo
sudo usermod -aG docker $USER && newgrp docker

# Bước 7: Verify
docker --version && docker run hello-world
⚠️ Gotcha macOS & Windows

macOS: Docker Desktop chạy một Linux VM nhỏ bên trong vì macOS kernel không hỗ trợ Linux containers native — file I/O mount từ /Users sẽ chậm hơn Linux native, đây là hành vi bình thường.
Windows: Luôn chạy Docker command trong WSL terminal (Ubuntu), không phải PowerShell/CMD. Volume mount trong WSL2 (/home/user/) nhanh hơn nhiều so với mount từ Windows path (/mnt/c/...).

4. Image và Container — Cặp Đôi Cốt Lõi

Hãy nghĩ về mối quan hệ này như: Image = Khuôn bánh (bất biến, immutable) · Container = Cái bánh được nặn ra (instance đang chạy). Từ 1 image có thể tạo nhiều container.

1
Docker Client gửi REST API
Lệnh docker run nginx:1.25 → Docker CLI gửi request đến Docker Daemon
2
Kiểm tra & Pull Image
Có image local? → dùng ngay. Không có → pull từ Docker Hub registry
3
Tạo Container Layer
Tạo writable layer mới trên đỉnh image layers (overlay filesystem)
4
Setup Namespaces & cgroups
Tạo PID, NET, MNT namespaces · cấu hình resource limits · setup iptables
5
Chạy ENTRYPOINT + CMD
Container PID 1 = process chính. Khi process này exit → container dừng.
bash — Lệnh vòng đời Container
# Chạy detached, đặt tên, map port, giới hạn tài nguyên
docker run -d --name my-nginx -p 8080:80 --memory="256m" --cpus="0.5" nginx:1.25

# Restart policies
docker run --restart=unless-stopped my-app    # ← KHUYẾN NGHỊ cho production
# no=default | always=restart kể cả khi stop | on-failure=chỉ khi crash

# Vào shell bên trong container
docker exec -it my-nginx /bin/bash

# Xem logs real-time
docker logs -f my-nginx --tail 100

# Xem CPU/RAM đang dùng
docker stats --no-stream

# Zombie process prevention (quan trọng với app fork processes)
docker run --init my-app    # tini làm PID 1 — reap zombie processes đúng cách

5. Dockerfile — Layer Caching & Multi-Stage Build

5.1 Tận dụng Layer Caching

Layer caching là chìa khóa để build nhanh. Docker cache mỗi layer — nếu input không đổi, Docker dùng lại từ cache. Quy tắc vàng: thứ ít thay đổi lên trước, thứ hay thay đổi xuống sau.

dockerfile — ❌ Kém vs ✅ Tối ưu
# ❌ Mất cache mỗi khi thay đổi 1 dòng code
FROM eclipse-temurin:21-jdk-alpine
WORKDIR /app
COPY . .                       # Invalidate TẤT CẢ cache khi đổi bất kỳ file
RUN ./mvnw package -DskipTests

# ✅ Tách dependencies (ít đổi) và source code (hay đổi)
FROM eclipse-temurin:21-jdk-alpine
WORKDIR /app
COPY pom.xml .                 # Step 1: chỉ copy manifest
COPY .mvn .mvn
COPY mvnw .
RUN ./mvnw dependency:go-offline -q  # Step 2: cache khi pom.xml không đổi!
COPY src ./src                 # Step 3: source thay đổi → chỉ invalidate từ đây
RUN ./mvnw package -DskipTests -q   # Step 4: build

Khi chỉ thay đổi code (không đổi pom.xml), Docker dùng lại cache step 1–2 → build nhanh hơn 5–10 lần!

5.2 Multi-Stage Build — Image Siêu Nhỏ

dockerfile — Multi-Stage (Java)
# === STAGE 1: BUILD (JDK + Maven + source code) ===
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /build
COPY pom.xml . && COPY .mvn/ .mvn/ && COPY mvnw .
RUN ./mvnw dependency:go-offline -q
COPY src ./src
RUN ./mvnw package -DskipTests -q

# === STAGE 2: RUNTIME (chỉ JRE + JAR — không có build tools!) ===
FROM alpine:3.19 AS runtime
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app
COPY --from=builder /build/target/myapp-*.jar app.jar

HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health | grep -q '"status":"UP"' || exit 1

USER appuser
EXPOSE 8080
# Exec form BẮT BUỘC — để SIGTERM truyền thẳng đến app (graceful shutdown)
ENTRYPOINT ["java", "-XX:+UseContainerSupport", "-XX:MaxRAMPercentage=75.0", "-jar", "app.jar"]
✅ Kết quả Multi-Stage

Không dùng multi-stage: ~600MB (JDK + Maven + source + JAR) · Dùng multi-stage: ~85MB (custom JRE + JAR) · Giảm ~86% kích thước!

Deep Dive
BuildKit Cache Mounts — Build nhanh hơn 10x trong CI/CD
Dành cho: Developer muốn tăng tốc pipeline CI/CD — cache Maven/npm giữa các lần build.
dockerfile — BuildKit cache mount
# syntax=docker/dockerfile:1    ← PHẢI là dòng ĐẦU TIÊN của Dockerfile!
FROM eclipse-temurin:21-jdk-alpine AS builder
WORKDIR /build
COPY pom.xml . && COPY .mvn/ .mvn/ && COPY mvnw .

# --mount=type=cache: giữ lại /root/.m2 giữa các lần build
# Lần đầu: download bình thường (~5 phút)
# Lần sau: dùng lại cache (~28 giây!) → nhanh hơn ~10 lần!
RUN --mount=type=cache,target=/root/.m2 \
    ./mvnw dependency:go-offline -q

COPY src ./src
RUN --mount=type=cache,target=/root/.m2 \
    ./mvnw package -DskipTests -q

# Tương tự cho Node.js:
# RUN --mount=type=cache,target=/root/.npm \
#     npm ci --prefer-offline
bash
docker buildx build .          # Luôn có BuildKit
docker builder du              # Xem cache đang chiếm bao nhiêu disk
docker builder prune -a        # Xóa build cache

6. Docker Networking — Multi-Network Architecture

Ứng dụng thực tế luôn cần nhiều component. Docker Networking cho phép chúng giao tiếp an toàn và có kiểm soát. Luôn dùng custom bridge network — không dùng default bridge (không có DNS, không có cách ly).

Pattern chuẩn: Multi-Network Architecture
🌐 Internet
↓
🔵 Frontend Network (public-facing)
Nginx (reverse proxy)
React App (static)
↕ Nginx giao tiếp cả 2 networks
🟢 Backend Network (internal only — không thể access từ Internet)
Spring Boot API
PostgreSQL
Redis
PostgreSQL và Redis KHÔNG thể access từ Internet. Nếu React app bị compromise, attacker không thể trực tiếp truy cập database.
yaml — docker-compose.yml multi-network
services:
  nginx:
    image: nginx:1.25-alpine
    networks: [frontend-network, backend-network]  # Nhận từ internet, forward tới app
    ports: ["80:80"]

  spring-app:
    build: ./backend
    networks: [backend-network]   # Chỉ backend — KHÔNG expose port ra ngoài!

  postgres:
    image: postgres:16-alpine
    networks: [backend-network]   # Nội bộ only

networks:
  frontend-network:
    name: myapp-frontend
  backend-network:
    name: myapp-backend

7. Docker Volumes — Persistent Data

Container là ephemeral (tạm thời) — data trong container layer sẽ mất khi container bị xóa. Volumes giải quyết vấn đề này.

Named Volume ✅
  • Docker quản lý hoàn toàn
  • Không mất khi container xóa
  • Dùng cho: DB, file upload
  • -v postgres-data:/var/lib/postgresql/data
Bind Mount
  • Mount thư mục từ host
  • Tốt cho hot-reload dev
  • Dùng cho: config, source code
  • -v $(pwd)/src:/app/src
tmpfs Mount
  • Lưu trong RAM
  • Mất khi container dừng
  • Dùng cho: session, temp secrets
  • --tmpfs /tmp:rw,size=100m
bash — Backup & Restore Volume
# BACKUP volume thành file tar.gz
docker run --rm \
  -v postgres-data:/data:ro \
  -v $(pwd)/backups:/backup \
  alpine \
  tar czf /backup/postgres-$(date +%Y%m%d-%H%M).tar.gz -C /data .

# RESTORE từ backup
docker volume create postgres-data-restored
docker run --rm \
  -v postgres-data-restored:/data \
  -v $(pwd)/backups:/backup \
  alpine \
  tar xzf /backup/postgres-20260501-1200.tar.gz -C /data

8. Docker Compose — Điều Phối Multi-Container Stack

Khi stack có nhiều services, bạn cần nhớ thứ tự khởi động, network, volumes, env vars cho từng service. Docker Compose giải quyết bằng một file YAML duy nhất.

yaml — Production-grade docker-compose.yml
# Docker Compose v2 — không cần trường "version:" (deprecated)
services:

  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: myapp_db
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: ${DB_PASSWORD}   # Đọc từ .env (KHÔNG commit git!)
    volumes:
      - postgres-data:/var/lib/postgresql/data
    networks: [backend-network]
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d myapp_db"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  app:
    build: { context: ., target: runtime }
    restart: unless-stopped
    environment:
      SPRING_DATASOURCE_URL: jdbc:postgresql://postgres:5432/myapp_db
      SPRING_DATASOURCE_PASSWORD: ${DB_PASSWORD}
    networks: [backend-network]
    depends_on:
      postgres:
        condition: service_healthy   # Chờ postgres healthy trước khi start app!
    deploy:
      resources:
        limits: { memory: 1G, cpus: '2.0' }

volumes:
  postgres-data:
networks:
  backend-network: { driver: bridge }
bash — Compose commands
docker compose up -d --build      # Start stack, build images
docker compose logs -f app        # Follow logs một service
docker compose ps                 # Xem trạng thái
docker compose exec app /bin/bash # Vào shell service
docker compose down               # Dừng và xóa containers (giữ volumes)
docker compose down -v            # Xóa luôn volumes (mất data!)

# Dev hot-reload (Compose 2.22+)
docker compose watch              # Auto sync code + rebuild khi file đổi

9. Docker Registry — Multi-Platform Build

Registry là kho lưu trữ Docker Image — như GitHub nhưng cho container images. Workflow chuẩn: Build → Push → Pull → Run.

bash — Docker Hub & Multi-Platform
# Push lên Docker Hub
docker login
docker tag myapp:1.0.0 yourusername/myapp:1.0.0
docker push yourusername/myapp:1.0.0

# Multi-platform build — fix exec format error giữa ARM64 (Mac M2) và AMD64 (server)
docker buildx create --name mybuilder --use
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t yourusername/myapp:1.0.0 \
  --push \
  .
# → Docker tự tạo "manifest list": server AMD64 pull AMD64, Mac M2 pull ARM64!
⚠️ Lỗi phổ biến từ khi Apple ra MacBook M1/M2

Dev build image trên MacBook M2 (ARM64) → push lên registry → server production pull về (AMD64) → exec /usr/local/bin/java: exec format error. Giải pháp dứt điểm: luôn dùng docker buildx build --platform linux/amd64,linux/arm64.

Docker giải quyết "works on my machine" — CI/CD giải quyết "deploy thủ công, hay quên bước". Kết hợp lại:

→

→
Push registry

yaml — .github/workflows/deploy.yml
name: Build, Test & Deploy
on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    services:                        # Service container cho integration tests
      postgres:
        image: postgres:16-alpine
        env: { POSTGRES_PASSWORD: testpass }
        options: --health-cmd pg_isready --health-interval 10s

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { java-version: '21', distribution: 'temurin', cache: maven }
      - run: ./mvnw test

  build-and-push:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }

    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with: { registry: ghcr.io, username: ${{ github.actor }}, password: ${{ secrets.GITHUB_TOKEN }} }
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PROD_HOST }}
          key: ${{ secrets.PROD_SSH_KEY }}
          script: |
            cd /opt/myapp
            docker compose pull app
            docker compose up -d --no-deps app  # Rolling update — zero downtime
            docker image prune -f

11. Security — Bảo Mật Docker Container

👤
Non-root user — Tạo user riêng với RUN addgroup && adduser và USER appuser trong Dockerfile. Nếu app bị tấn công, attacker chỉ có quyền user, không phải root.
🔒
no-new-privileges — --security-opt no-new-privileges:true ngăn process trong container leo thang quyền qua setuid/setgid.
🛡️
Drop capabilities — --cap-drop=ALL --cap-add=NET_BIND_SERVICE loại bỏ ~14 Linux capabilities mặc định, chỉ giữ lại những gì thực sự cần.
🔢
PID limit — --pids-limit=100 ngăn fork bomb attack (process liên tục fork chính nó cho đến khi cạn kiệt PID table).
📂
Read-only filesystem — --read-only --tmpfs /tmp:rw,size=100m ngăn attacker ghi file độc hại vào container.
🔑
Không hardcode secrets — ENV PASSWORD=xxx và ARG API_KEY=xxx lộ trong docker history! Dùng runtime env var, Docker Secrets, hoặc external Secret Manager.
🔍
Scan image — docker scout cves myapp:1.0.0 hoặc trivy image myapp:1.0.0 trước khi push production. Tích hợp vào CI/CD pipeline.

12. Maintenance — Dọn Dẹp và Giám Sát

bash — Disk cleanup & monitoring
# Xem Docker đang dùng bao nhiêu disk
docker system df -v

# Dọn dẹp thủ công (kiểm soát chính xác)
docker container prune          # Xóa containers đã dừng
docker image prune -a           # Xóa images không dùng
docker volume prune             # ⚠️ Cẩn thận: xóa data!
docker builder prune -a         # Xóa build cache

# Nuclear option (dọn sạch tất cả)
docker system prune -a          # Không xóa volumes

# Tự động dọn dẹp hàng ngày lúc 3 AM
# Thêm vào crontab: 0 3 * * * docker system prune -f >> /var/log/docker-prune.log

# Xem resource real-time
docker stats --format "table {{.Name}}\t{{.CPUPerc}}\t{{.MemUsage}}\t{{.NetIO}}"

14. Troubleshooting — Xử Lý Khi Mọi Thứ Đi Sai

Triệu chứng
Container dừng ngay sau khi start
Debug & Fix
docker logs <id> · docker inspect <id> --format='{{.State.ExitCode}}' · Chạy docker run --entrypoint /bin/sh -it myapp để vào shell debug
Triệu chứng
exec format error khi run
Debug & Fix
docker inspect myapp --format='{{.Architecture}}' · Image ARM64 chạy trên AMD64 (hoặc ngược lại) · Fix: docker buildx build --platform linux/amd64,linux/arm64 --push
Triệu chứng
OOMKilled — container chết đột ngột
Debug & Fix
docker inspect <id> --format='{{.State.OOMKilled}}' = true · Tăng --memory limit · Với Java: thêm -XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0
Triệu chứng
Containers không connect được nhau
Debug & Fix
docker network inspect my-network → kiểm tra cả hai container có trong "Containers" không · docker exec -it myapp ping postgres · docker exec -it myapp nc -zv postgres 5432
Triệu chứng
429 Too Many Requests (Docker Hub)
Debug & Fix
Rate limit: anonymous 100 pulls/6h · Fix: docker login trong CI/CD · Hoặc dùng ghcr.io làm mirror · Cache layers với cache-from: type=gha
Triệu chứng
Compose khởi động không đúng thứ tự
Debug & Fix
Thêm healthcheck vào service dependency và dùng depends_on: condition: service_healthy · docker compose config để debug biến môi trường
Triệu chứng
Permission denied khi mount volume
Debug & Fix
docker run -it myapp ls -la /app/ → xem owner · Fix: sudo chown -R 1000:1000 /host/data · Hoặc dùng Named Volume thay vì Bind Mount

15. Kết Luận & Lộ Trình Học Tiếp

Key Takeaways

  • 🐳 Container = Linux namespaces + cgroups + overlay FS — nhẹ, nhanh, cách ly mà không cần VM
  • 📝 Dockerfile = layer caching + multi-stage + BuildKit cache mounts + Distroless → image nhỏ, build nhanh
  • 🌐 Networking = multi-network architecture — database không accessible từ internet
  • 💾 Volumes = persistent data, không mất khi container xóa, backup/restore dễ dàng
  • 🎼 Compose = điều phối multi-container, compose watch cho dev hot-reload
  • 📦 Registry = multi-platform builds (arm64+amd64) tránh exec format error
  • 🚀 CI/CD = GitHub Actions build → scan → push → deploy, zero-downtime, rollback strategy
  • 🔒 Security = non-root, no-new-privileges, pids-limit, Distroless, Docker Secrets
  • 🔧 Troubleshoot = exec format error, OOMKilled, rate limit 429, permissions — có công thức debug rõ ràng

Lộ trình tiếp theo

Bây giờ
Docker thuần nâng cao: Docker Swarm · BuildKit attestations · Content Trust (DCT) · Docker Extensions
Tiếp theo
Container Orchestration: Kubernetes (Pods, Deployments, Services, Ingress, HPA) · Helm · Kustomize · Managed K8s (EKS, GKE, AKS)
Sau đó
Production & Observability: Prometheus + Grafana · ELK Stack / Grafana Loki · Jaeger · OpenTelemetry · Service Mesh (Istio, Linkerd)
Câu hỏi tự kiểm tra

Trả lời được những câu này → bạn sẵn sàng dùng Docker trong production:

  1. Container và VM khác nhau thế nào? Linux namespaces và cgroups đóng vai trò gì?
  2. Tại sao nên COPY pom.xml trước khi COPY src? Giải thích layer caching.
  3. Multi-stage build giúp gì? Kết quả kích thước image thay đổi như thế nào?
  4. BuildKit cache mount (--mount=type=cache) giải quyết vấn đề gì? Nhanh hơn bao nhiêu lần?
  5. Tại sao ENTRYPOINT ["java", "-jar"] tốt hơn ENTRYPOINT java -jar?
  6. Trong multi-network architecture, tại sao PostgreSQL không nên ở cùng network với Nginx?
  7. Khi nào dùng Named Volume, khi nào dùng Bind Mount, khi nào dùng tmpfs?
  8. depends_on: condition: service_healthy hoạt động thế nào? Cần điều kiện gì?
  9. Phân biệt .env file, Docker Secrets, và External Secret Manager. Khi nào dùng cái nào?
  10. Distroless image là gì? Ưu và nhược điểm so với Alpine khi debugging?
  11. Lỗi exec format error xảy ra khi nào? Cách fix dứt điểm là gì?
  12. Container bị OOMKilled — cách debug và fix với Java app?

Trả lời được 9/12 → bạn solid ở Docker fundamentals và production basics.
