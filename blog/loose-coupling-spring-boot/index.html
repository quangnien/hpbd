1. Giới thiệu — "Fix 1 bug, sinh ra 10 bug"

Bạn đã bao giờ gặp tình huống này chưa?

Fix xong bug A, bỗng dưng bug B, C, D, E xuất hiện từ đâu không biết.

Mình gọi đó là impact blast — sửa một chỗ nhỏ mà ảnh hưởng khắp nơi. Nói ví von thì cái nhà (hệ thống) chúng ta xây lên, chỉ cần ai đó rung nhẹ là sập nguyên căn. Đây là vấn đề kinh điển của những hệ thống bị tight coupling (phụ thuộc chặt chẽ).

Hay một tình huống thực tế hơn trong môi trường Agile:

Sprint 3, PO yêu cầu đổi database từ MySQL sang Google Cloud Spanner. Tưởng đơn giản, nhưng bạn phát hiện mình phải sửa 47 class vì chúng đều đang new MySQLRepository() trực tiếp.

Loose Coupling (phụ thuộc lỏng lẻo) là giải pháp cho cả hai tình huống trên. Khái niệm này dễ hiểu nhưng khó đạt được trong thực tế. Trong bài viết này, chúng ta sẽ tập trung vào Loose Coupling trong OOP — cụ thể là việc quản lý dependency (phụ thuộc) giữa các class, với code mẫu bằng Java 17+ và Spring Boot 3.x.

2. Tight Coupling vs Loose Coupling

Tight Coupling (phụ thuộc chặt chẽ)

Khi một class phụ thuộc trực tiếp vào class khác. Bất kỳ thay đổi nào ở class bị phụ thuộc đều có thể gây vỡ class đang dùng nó.

Loose Coupling (phụ thuộc lỏng lẻo)

Khi một class chỉ phụ thuộc vào abstraction (interface hoặc lớp trừu tượng), không phụ thuộc vào implementation cụ thể. Thay đổi bên trong implementation không ảnh hưởng đến class đang dùng.

So sánh trực quan

❌ Tight Coupling
UserService ──new──→ MySQLUserRepository
Hard dependency — không thể thay thế
✅ Loose Coupling
UserService ──→ «interface»
UserRepository
MySQLUserRepository
SpannerUserRepository
MongoUserRepository

Bạn thấy sự khác biệt chứ? Với tight coupling, UserService bị "dính chặt" vào MySQLUserRepository. Muốn đổi sang Spanner hay MongoDB? Phải sửa UserService. Có 50 class đang new MySQLUserRepository()? Phải sửa cả 50.

Với loose coupling, UserService chỉ biết interface UserRepository — muốn đổi database chỉ cần thêm một implementation mới, không đụng vào UserService.

Ví dụ Tight Coupling

Java 
public class UserService {

    // ❌ Hard dependency — đây chính là nguồn gốc tight coupling
    // Từ khóa "new" gắn chặt UserService với MySQLUserRepository
    private final MySQLUserRepository userRepository = new MySQLUserRepository();

    public String getById(int id) {
        return userRepository.getById(id);
    }
}

Bạn nhìn thấy từ khóa new không? Đó chính là hard dependency. UserService đang phụ thuộc cứng vào MySQLUserRepository.

Hậu quả của tight coupling:

Tình huốngCông việc cần làm
Đổi tên MySQLUserRepositoryPhải sửa UserService và TẤT CẢ class dùng nó
Đổi constructor params của repositoryPhải sửa từng chỗ gọi new
Muốn chuyển sang Spanner/MongoDBPhải sửa hàng chục class
Muốn viết unit testKhông thể mock MySQLUserRepository (class cụ thể)
CI/CD cần chạy test độc lập với DBGần như bất khả thi

3.1. IoC — Inversion of Control (Đảo ngược sự điều khiển)

IoC là gì?

IoC là một design principle (nguyên lý thiết kế) — dùng để đảo ngược ai là người "điều khiển" việc tạo và quản lý object.

Ví dụ đời thực: Bạn lái xe đi làm mỗi ngày — bạn đang tự điều khiển xe. Áp dụng IoC nghĩa là bạn gọi taxi — người khác lái cho bạn. Bạn chỉ cần quan tâm đích đến, không cần quan tâm cách lái hay xe đó là Toyota hay Honda. Đó là đảo ngược sự điều khiển.

Áp dụng Factory Pattern để implement IoC:

Java 
public class UserService {

    private final MySQLUserRepository userRepository;

    public UserService() {
        // ✅ Bước 1: Đảo ngược điều khiển — Factory sẽ lo việc tạo object
        // UserService không còn tự "new" nữa
        this.userRepository = UserRepositoryFactory.create();
    }

    public String getById(int id) {
        return userRepository.getById(id);
    }
}

public class UserRepositoryFactory {

    // Factory chịu trách nhiệm tạo object — UserService không cần biết chi tiết
    public static MySQLUserRepository create() {
        return new MySQLUserRepository();
    }
}

UserService không còn tự tạo MySQLUserRepository nữa — việc đó đã được đẩy về Factory. Đây chính là IoC.

⚠️ Lưu ý

Tuy đã áp dụng IoC, UserService vẫn phụ thuộc vào kiểu MySQLUserRepository ở dòng khai báo field, và còn phụ thuộc thêm vào UserRepositoryFactory. Đây chưa phải loose coupling — chỉ là bước trung gian!

Deep Dive
Spring ApplicationContext hoạt động thế nào bên trong?
Dành cho: Intermediate / Advanced developer muốn hiểu "phép màu" của Spring không phải phép màu.

Khi bạn chạy một Spring Boot application, câu lệnh đầu tiên là:

Java 
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        // Đây chính là khoảnh khắc IoC Container được tạo
        SpringApplication.run(MyApplication.class, args);
    }
}

Bên dưới hood, SpringApplication.run() thực hiện 4 giai đoạn chính:

1
Bean Definition Scanning
  • Spring quét tất cả class có @Component, @Service, @Repository, @Controller
  • Đọc tất cả @Configuration class và @Bean method
  • Tạo BeanDefinition — bản mô tả, chưa phải object thật
2
Dependency Resolution
  • Xây dựng dependency graph giữa các bean
  • Phát hiện circular dependency tại đây — sẽ throw exception
  • Xác định thứ tự khởi tạo bean
3
Bean Instantiation
  • Tạo object theo đúng thứ tự dependency
  • Inject dependency qua constructor (preferred) hoặc setter
4
Post-Processing
  • Gọi @PostConstruct methods
  • Áp dụng BeanPostProcessor (AOP, validation, etc.)
  • ApplicationContext sẵn sàng phục vụ

Tại sao constructor injection tốt hơn field injection?

Với constructor injection, dependency được truyền vào tại thời điểm object được tạo — immutable ngay từ đầu, không cần thao tác phụ. Với field injection (@Autowired trên field), Spring phải dùng Java Reflection để ghi vào private field sau khi object đã được tạo — phá vỡ encapsulation và làm cho object tồn tại trong trạng thái "nửa chừng" (partially initialized):

Java 
// Spring làm gì với field injection (phía sau hậu trường):
Field field = UserService.class.getDeclaredField("userRepository");
field.setAccessible(true); // ❌ Phá vỡ encapsulation — private field không còn private!
field.set(userServiceInstance, userRepositoryBean);
// → Object tồn tại trong khoảnh khắc userRepository = null trước dòng này

Khoảng thời gian "nửa chừng" này là nguồn gốc của NullPointerException khó debug, và lý do field injection không thread-safe trong quá trình khởi tạo đa luồng. Đây là lý do Spring chính thức khuyến khích constructor injection từ Spring 4.3+.

3.2. DIP — Dependency Inversion Principle (Nguyên lý đảo ngược phụ thuộc)

DIP là chữ D trong SOLID:

1. High-level modules không nên phụ thuộc vào low-level modules. Cả hai nên phụ thuộc vào abstraction.
2. Abstraction không nên phụ thuộc vào details. Details nên phụ thuộc vào abstraction.

Áp dụng vào ví dụ:

Khái niệm DIPTương ứng trong code
High-level moduleUserService — chứa business logic
Low-level moduleMySQLUserRepository — chứa persistence logic
AbstractionUserRepository — interface (chưa có trong code ban đầu!)
UserService High-level module
──→
«interface» UserRepository Abstraction
↑ ↑
MySQLUserRepositoryLow-level
SpannerUserRepositoryLow-level
Cả hai đều phụ thuộc vào abstraction — không ai phụ thuộc trực tiếp vào nhau

Cả high-level và low-level module đều hướng về phía abstraction — đó là "đảo ngược phụ thuộc".

Java 
// ✅ Bước 1: Tạo interface — đây chính là abstraction
public interface UserRepository {
    String getById(int id);
    void save(String userData);
}

// ✅ Bước 2: MySQLUserRepository implements interface (detail phụ thuộc abstraction)
public class MySQLUserRepository implements UserRepository {
    @Override
    public String getById(int id) {
        // Logic truy vấn MySQL
        return "user data from MySQL";
    }

    @Override
    public void save(String userData) {
        // Logic lưu vào MySQL
    }
}

// ✅ Bước 3: UserService phụ thuộc vào interface, KHÔNG phải class cụ thể
public class UserService {

    // Kiểu là UserRepository (interface), không phải MySQLUserRepository (class)
    private final UserRepository userRepository;

    public UserService() {
        // Factory bây giờ trả về interface — không cần biết implementation cụ thể
        this.userRepository = UserRepositoryFactory.create();
    }

    public String getById(int id) {
        return userRepository.getById(id);
    }
}

public class UserRepositoryFactory {

    // ✅ Return type là interface — đây là điểm mấu chốt của DIP
    public static UserRepository create() {
        return new MySQLUserRepository();
    }
}

Chỉ thay đổi một chút mà đã thỏa mãn cả 2 nguyên lý của DIP:

  • UserService phụ thuộc vào UserRepository (interface) — không phải MySQLUserRepository
  • MySQLUserRepository phụ thuộc vào UserRepository (implements interface)
  • Interface UserRepository không biết gì về MySQLUserRepository
⚠️ Vẫn còn vấn đề

UserService vẫn phụ thuộc vào UserRepositoryFactory. Nếu Factory thay đổi, UserService vẫn phải sửa theo. Chúng ta cần bước tiếp theo — DI.

3.3. DI — Dependency Injection (Tiêm phụ thuộc)

DI là gì?

DI là một design pattern — đơn giản là inject (tiêm) dependency vào class từ bên ngoài, thay vì để class tự tạo bên trong. Class không cần biết ai cung cấp dependency — chỉ cần nhận và dùng.

3 kiểu inject:

LoạiCú phápKhuyến nghị
Constructor injectionInject qua constructor✅ Best practice — dependency luôn sẵn có, dùng được final, dễ test
Setter injectionInject qua setter method⚠️ Dùng khi dependency là optional hoặc có thể thay đổi sau khi khởi tạo
Field injection@Autowired trực tiếp trên field❌ Anti-pattern — không dùng được final, khó test, phá vỡ encapsulation

Ví dụ 3 kiểu inject cạnh nhau:

Java 
// ✅ Constructor injection — RECOMMENDED
@Service
public class OrderService {
    private final UserRepository userRepository; // final ✅

    public OrderService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

// ⚠️ Setter injection — chỉ dùng khi dependency là optional
@Service
public class NotificationService {
    private EmailClient emailClient; // không final — có thể null nếu không inject

    // Spring gọi setter này nếu EmailClient bean tồn tại; bỏ qua nếu không có
    @Autowired(required = false)
    public void setEmailClient(EmailClient emailClient) {
        this.emailClient = emailClient;
    }

    public void notify(String message) {
        if (emailClient != null) { // phải null-check vì có thể chưa inject
            emailClient.send(message);
        }
    }
}

// ❌ Field injection — ANTI-PATTERN, tránh dùng
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository; // không final, khó test, ẩn dependency
}

Áp dụng Constructor Injection:

Java 
public class UserService {

    // ✅ final — đảm bảo dependency không bị thay đổi sau khi inject
    private final UserRepository userRepository;

    // ✅ Dependency được inject từ bên ngoài qua constructor
    // UserService không biết và không cần biết ai cung cấp — MySQL hay Spanner hay Mock
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public String getById(int id) {
        return userRepository.getById(id);
    }
}

Bạn thấy không? Không còn new, không còn Factory bên trong UserService.

Bây giờ kiểm tra lại:

  • Muốn đổi sang SpannerUserRepository → UserService không cần thay đổi gì
  • Muốn viết unit test → new UserService(mockRepository) là xong
  • Đổi constructor params của MySQLUserRepository → UserService không hề biết
✅ Kết hợp DI + DIP → Chúng ta đã đạt được Loose Coupling!

Nhưng chờ đã — ai sẽ là người tạo và inject dependency? Nếu dự án có hàng trăm class, việc tự tay new và inject sẽ trở thành cơn ác mộng. Đó là lý do chúng ta cần IoC Container.

Deep Dive
Field Injection — Tại sao là anti-pattern?
Dành cho: Developer đang còn dùng @Autowired trên field và muốn hiểu tại sao nên bỏ.

Rất nhiều developer mới học Spring thường viết như thế này:

Java 
@Service
public class UserService {

    @Autowired // ❌ Field injection — đây là anti-pattern
    private UserRepository userRepository;
}

Trông có vẻ tiện, nhưng thực ra nó có nhiều vấn đề nghiêm trọng:

Vấn đề 1: Không thể dùng final

Java 
// ❌ Compile error — không thể inject vào final field qua @Autowired
@Autowired
private final UserRepository userRepository; // ← lỗi ngay tại đây

Không có final → dependency có thể bị reassign sau khi khởi tạo → race condition trong môi trường multi-thread.

Vấn đề 2: Không thể test nếu không có Spring Context

Java 
// ❌ Trong unit test, userRepository sẽ là null!
UserService service = new UserService(); // Không có cách nào inject dependency
service.getById(1); // NullPointerException

// ✅ Với constructor injection: dễ dàng test không cần Spring
UserRepository mockRepo = Mockito.mock(UserRepository.class);
UserService service = new UserService(mockRepo); // inject thẳng

Vấn đề 3: Ẩn dependency — vi phạm nguyên tắc "explicit is better than implicit"

Nhìn vào class dùng field injection, bạn không thể biết ngay nó cần những gì để hoạt động — phải đọc toàn bộ body class mới thấy các @Autowired. Nhìn vào constructor injection, dependencies hiện rõ ngay ở chữ ký constructor.

✅ Kết luận

Từ Spring 4.3+, nếu class chỉ có duy nhất 1 constructor, Spring tự động inject mà không cần @Autowired. Đây là best practice hiện tại — loại bỏ hoàn toàn @Autowired trên field.

3.4. IoC Container — "Cái hộp thần kỳ"

Ở trên mọi thứ đã loose coupling, nhưng khi thực tế sử dụng thì sao? Ai sẽ gọi constructor và truyền dependency vào?

Java 
// Ở đâu đó trong code, vẫn phải có người làm điều này:
UserService userService = new UserService(new MySQLUserRepository());

Với 1–2 dependency thì ổn. Nhưng thực tế production, một use case có thể phụ thuộc 5, 7 hoặc nhiều hơn dependency:

Java — Thực tế production (không dùng IoC Container) 
// Thực tế production trông như thế này (không dùng IoC Container):
ViewPluginUseCase useCase = new ViewPluginUseCase(
    new PluginService(
        new MySQLPluginRepository(),
        new MySQLUserRepository()
    ),
    new InMemoryFileRepository(),
    new MemberListService(new MySQLUserRepository()), // MySQLUserRepository tạo 2 lần!
    new TargetService(
        new TargetRepository(),
        new MemberListService(new MySQLUserRepository()) // tạo lần 3!
    ),
    new ConfigRepository(),
    new ParamRepository()
);
⚠️ Vấn đề

MySQLUserRepository và MemberListService bị tạo nhiều lần — lãng phí tài nguyên, và mỗi instance là một object riêng biệt (không phải singleton).

IoC Container giải quyết toàn bộ vấn đề trên. Nó là một framework tự động:

  1. Scan và phát hiện tất cả các class cần quản lý
  2. Tạo object đúng một lần (singleton mặc định)
  3. Phân tích dependency graph và xác định thứ tự khởi tạo
  4. Inject tất cả dependency đúng chỗ
  5. Quản lý lifecycle — khởi tạo, dùng, hủy
❌ Không có IoC Container
Developer → tự new → tự inject → tự quản lý lifecycle → 😵 đau đầu
✅ Có IoC Container
Developer → khai báo @Service → Container lo hết 🎉
Container → scan → tạo bean → inject → lifecycle

Spring Boot — IoC Container thực tế:

Java 
// ✅ Khai báo đây là bean — Spring Container sẽ quản lý lifecycle
@Repository
public class MySQLUserRepository implements UserRepository {
    @Override
    public String getById(int id) {
        return "user from MySQL";
    }
}

// ✅ Spring tự phát hiện UserRepository → tìm bean implement nó → inject vào constructor
@Service
public class UserService {

    private final UserRepository userRepository;

    // Spring 4.3+: chỉ 1 constructor → tự inject, không cần @Autowired
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public String getById(int id) {
        return userRepository.getById(id);
    }
}

Không một chữ new nào trong business logic. Spring lo hết.

Deep Dive
Bean Scope & Lifecycle trong Spring
Dành cho: Developer muốn hiểu Spring Container quản lý object như thế nào.

Spring hỗ trợ nhiều Bean Scope (phạm vi sống của bean):

ScopeMô tảDùng khi nào
singletonMặc định — 1 instance cho toàn bộ ApplicationContextService, Repository, Controller — hầu hết trường hợp
prototypeTạo instance mới mỗi lần injectObject có state thay đổi giữa các lần dùng
request1 instance / HTTP request (chỉ trong web app)Object lưu data của 1 request cụ thể
session1 instance / HTTP sessionObject lưu data của 1 user session

Singleton vs Prototype — hiểu đúng để tránh bug:

Java 
// ✅ Singleton scope (mặc định) — dùng cho stateless service
@Service // = @Service + @Scope("singleton")
public class UserService {
    // ⚠️ KHÔNG được có mutable state ở đây!
    // Vì chỉ có 1 instance dùng chung cho tất cả requests
    private final UserRepository userRepository; // ✅ final, immutable

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

// ⚠️ Đây là BUG phổ biến với singleton:
@Service
public class BadUserService {
    private List<String> requestLog = new ArrayList<>(); // ❌ Mutable state trong singleton!
    // Instance này dùng chung → requestLog sẽ bị tích lũy từ TẤT CẢ requests
    // Thread-unsafe, memory leak
}

Bean Lifecycle — vòng đời đầy đủ:

Java 
import lombok.extern.slf4j.Slf4j;
import javax.sql.DataSource;
import java.sql.Connection;
import java.sql.SQLException;

@Slf4j // ✅ Lombok tự tạo field "log" — không cần khai báo thủ công
@Service
public class DatabaseConnectionService {

    // ✅ DataSource được inject qua constructor — không tự khởi tạo
    private final DataSource dataSource;

    // ⚠️ Ngoại lệ có kiểm soát: connection không thể khai báo final
    // vì nó được khởi tạo trong @PostConstruct, không phải trong constructor.
    // Tuy nhiên, field này chỉ được gán đúng 1 lần và không bao giờ bị reassign
    // sau đó — "effectively immutable after startup". Pattern này chấp nhận được
    // khi resource cần dependency để khởi tạo (DataSource chưa ready ở constructor).
    private Connection connection;

    public DatabaseConnectionService(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    // ✅ @PostConstruct — gọi SAU KHI Spring inject xong tất cả dependency
    @PostConstruct
    public void init() throws SQLException {
        this.connection = dataSource.getConnection();
        log.info("Database connection initialized successfully");
    }

    // ✅ @PreDestroy — gọi TRƯỚC KHI Spring hủy bean (khi app shutdown)
    @PreDestroy
    public void cleanup() {
        if (connection != null) {
            try {
                connection.close();
                log.info("Database connection closed");
            } catch (SQLException e) {
                log.warn("Failed to close database connection", e);
            }
        }
    }
}
💡 Tip quan trọng

Không bao giờ inject prototype scope bean vào singleton scope bean qua constructor — Spring chỉ inject 1 lần, prototype bean sẽ hoạt động như singleton. Dùng ObjectProvider<T> hoặc ApplicationContext.getBean() nếu cần tạo prototype mới mỗi lần.

4. Ví dụ thực tế với Spring Boot

Trong Spring Boot, ApplicationContext chính là hiện thực hóa của IoC Container — nó scan, tạo, và quản lý toàn bộ bean trong ứng dụng. Toàn bộ ví dụ dưới đây là production-grade code, không có từ khóa new trong business logic.

Interface tầng Service — Áp dụng DIP

Java 
// ✅ Abstraction — module khác chỉ phụ thuộc vào interface này
public interface SecurityCodeFactoryService {

    /**
     * Tạo security code dựa trên thông tin card.
     *
     * @param request thông tin card đầu vào (đã validate tại Controller)
     * @return thông tin card kèm security code đã tạo
     */
    CardInfoResponse generateSecurityCode(CardInfoRequest request);
}

Implementation — Tầng Service (Detail phụ thuộc abstraction)

Java 
@Service // ✅ Đăng ký với Spring Container — mặc định là singleton
public class DynamicSecurityCodeFactoryServiceImpl
        implements SecurityCodeFactoryService {

    private final AmaorRestClient amaorRestClient;

    // ⚠️ @Value KHÔNG thể dùng với final field!
    // Spring inject @Value SAU khi constructor chạy xong (qua reflection),
    // còn final field phải được gán ngay trong constructor → xung đột.
    // Workaround: dùng @ConfigurationProperties hoặc inject String qua constructor.
    @Value("${contract-security-api.apikey}")
    private String apiKey;

    // ✅ Constructor injection — Spring 4.3+: không cần @Autowired với 1 constructor
    public DynamicSecurityCodeFactoryServiceImpl(AmaorRestClient amaorRestClient) {
        this.amaorRestClient = amaorRestClient;
    }

    @Override
    public CardInfoResponse generateSecurityCode(CardInfoRequest request) {
        return amaorRestClient.fetchSecurityCode(request, apiKey);
    }
}
💡 Tip nâng cao — inject @Value vào final field

Nếu muốn apiKey là final, hãy chuyển sang inject qua constructor:

Java 
public DynamicSecurityCodeFactoryServiceImpl(
        AmaorRestClient amaorRestClient,
        @Value("${contract-security-api.apikey}") String apiKey) {
    this.amaorRestClient = amaorRestClient;
    this.apiKey = apiKey; // ✅ Giờ apiKey có thể khai báo final
}

Hoặc dùng @ConfigurationProperties để nhóm toàn bộ config vào 1 record/class — đây là best practice cho production.

Controller — Orchestration Layer

Java 
@RestController
@RequestMapping("/v1/api/security-code")
public class SecurityCodeController {

    // ✅ Phụ thuộc vào interface, không phải concrete class
    private final SecurityCodeFactoryService securityCodeService;
    private final NotificationService notificationService;

    // ✅ Constructor injection — Spring tự động inject tất cả dependency
    // Không một chữ "new" nào ở đây!
    public SecurityCodeController(
            SecurityCodeFactoryService securityCodeService,
            NotificationService notificationService) {
        this.securityCodeService = securityCodeService;
        this.notificationService = notificationService;
    }

    @PostMapping("/generate")
    public ResponseEntity<CardInfoResponse> generate(
            @Valid @RequestBody CardInfoRequest request) {
        // Controller chỉ orchestrate — không chứa business logic
        CardInfoResponse response = securityCodeService.generateSecurityCode(request);
        notificationService.notifyGenerated(response);
        return ResponseEntity.ok(response);
    }
}

@Qualifier và @Primary — Chọn implementation khi có nhiều class cùng implement 1 interface

Khi một interface có nhiều implementation, Spring sẽ không biết inject cái nào và sẽ throw NoUniqueBeanDefinitionException. Có 2 cách giải quyết:

Cách 1: @Qualifier — chỉ định rõ tên bean cần inject:

Java 
// ✅ Đặt tên rõ ràng cho từng implementation
@Service("dynamic")
public class DynamicSecurityCodeServiceImpl implements SecurityCodeFactoryService { ... }

@Service("static")
public class StaticSecurityCodeServiceImpl implements SecurityCodeFactoryService { ... }

@Service("custom")
public class CustomSecurityCodeServiceImpl implements SecurityCodeFactoryService { ... }

// Khi inject — dùng @Qualifier để chỉ định chính xác bean nào
@RestController
public class SecurityCodeFactoryController {

    private final SecurityCodeFactoryService dynamicService;
    private final SecurityCodeFactoryService staticService;
    private final SecurityCodeFactoryService customService;

    public SecurityCodeFactoryController(
            @Qualifier("dynamic") SecurityCodeFactoryService dynamicService,
            @Qualifier("static") SecurityCodeFactoryService staticService,
            @Qualifier("custom") SecurityCodeFactoryService customService) {
        this.dynamicService = dynamicService;
        this.staticService = staticService;
        this.customService = customService;
    }
}

Cách 2: @Primary — đánh dấu 1 implementation làm mặc định:

Java 
@Primary // ✅ Implementation mặc định khi không có @Qualifier
@Service
public class DynamicSecurityCodeServiceImpl implements SecurityCodeFactoryService { ... }

@Service("custom")
public class CustomSecurityCodeServiceImpl implements SecurityCodeFactoryService { ... }

@RestController
@RequestMapping("/v1/api/default")
public class DefaultController {

    private final SecurityCodeFactoryService securityCodeService;

    public DefaultController(SecurityCodeFactoryService securityCodeService) {
        // Spring inject DynamicSecurityCodeServiceImpl vì nó có @Primary
        this.securityCodeService = securityCodeService;
    }
}
✅ Quy tắc chọn

Dùng @Primary khi có 1 implementation "chính" và các implementation khác chỉ dùng trong trường hợp đặc biệt. Dùng @Qualifier khi cùng lúc cần inject nhiều implementation khác nhau.

@Bean — Factory Pattern trong thế giới Spring

Spring cung cấp @Bean trong @Configuration để tạo object phức tạp — tương tự Factory Pattern nhưng được Container quản lý. Dùng khi cần tạo object từ thư viện bên ngoài (không thể thêm annotation trực tiếp vào source).

Java 
@Configuration
public class InfrastructureConfig {

    // ✅ Tạo HikariCP DataSource — thư viện bên ngoài, không sửa được source
    @Bean
    public DataSource dataSource(
            @Value("${spring.datasource.url}") String url,
            @Value("${spring.datasource.username}") String username,
            @Value("${spring.datasource.password}") String password) {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl(url);
        config.setUsername(username);
        config.setPassword(password);
        config.setMaximumPoolSize(10);
        return new HikariDataSource(config);
    }

    // ✅ Tạo RestClient để gọi external API
    @Bean
    public RestClient amaorRestClient(
            @Value("${amaor.base-url}") String baseUrl) {
        return RestClient.builder()
                .baseUrl(baseUrl)
                .defaultHeader("Content-Type", "application/json")
                .build();
    }
}

Circular Dependency — Cái bẫy hay gặp nhất
Dành cho: Developer từng gặp BeanCurrentlyInCreationException và không hiểu tại sao.

Circular dependency là gì?

ServiceA
───→ cần
ServiceB
───→ cần
ServiceA
🔄 Vòng tròn bất tận — con gà vs quả trứng, Spring không biết tạo cái nào trước
Java — ❌ Circular dependency 
@Service
public class OrderService {
    private final PaymentService paymentService; // OrderService cần PaymentService

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}

@Service
public class PaymentService {
    private final OrderService orderService; // PaymentService cần OrderService → vòng tròn!

    public PaymentService(OrderService orderService) {
        this.orderService = orderService;
    }
}
Error 
The dependencies of some of the beans in the application context form a cycle:
orderService → paymentService → orderService

3 cách giải quyết (theo thứ tự ưu tiên):

Giải pháp 1 (tốt nhất): Tách logic dùng chung sang class thứ 3

Java — ✅ Mediator Pattern 
// ✅ Tạo class trung gian chứa logic phối hợp
@Service
public class OrderPaymentMediator {

    private final OrderRepository orderRepository;
    private final PaymentRepository paymentRepository;

    public OrderPaymentMediator(
            OrderRepository orderRepository,
            PaymentRepository paymentRepository) {
        this.orderRepository = orderRepository;
        this.paymentRepository = paymentRepository;
    }

    public void processOrderPayment(Long orderId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new IllegalArgumentException("Order not found: " + orderId));
        paymentRepository.createPaymentFor(order);
    }
}

// ✅ OrderService giờ chỉ biết OrderRepository — không phụ thuộc PaymentService
@Service
public class OrderService {
    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}

// ✅ PaymentService giờ chỉ biết PaymentRepository
@Service
public class PaymentService {
    private final PaymentRepository paymentRepository;

    public PaymentService(PaymentRepository paymentRepository) {
        this.paymentRepository = paymentRepository;
    }
}

Giải pháp 2: Dùng Event-Driven (ApplicationEventPublisher)

Java — ✅ Event-Driven 
// ✅ Định nghĩa Event class
public class OrderCreatedEvent extends ApplicationEvent {

    private final Order order;

    public OrderCreatedEvent(Order order) {
        super(order);
        this.order = order;
    }

    public Order getOrder() { return order; }
}

// ✅ OrderService publish event thay vì gọi trực tiếp PaymentService
@Service
public class OrderService {

    private final ApplicationEventPublisher eventPublisher;

    public OrderService(ApplicationEventPublisher eventPublisher) {
        this.eventPublisher = eventPublisher;
    }

    public void createOrder(Order order) {
        eventPublisher.publishEvent(new OrderCreatedEvent(order));
    }
}

@Service
public class PaymentService {

    @EventListener
    public void handleOrderCreated(OrderCreatedEvent event) {
        // Xử lý payment khi nhận được event
    }
}

Giải pháp 3 (tránh dùng): @Lazy injection

Java — ⚠️ Quick fix 
// ⚠️ @Lazy phải đặt trên constructor parameter — Spring tạo proxy tại đây
@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(@Lazy PaymentService paymentService) {
        this.paymentService = paymentService;
    }
}
⚠️ Lưu ý

@Lazy là "quick fix" — dùng để giải quyết tạm thời. Circular dependency thường là triệu chứng của thiết kế chưa tốt — hãy refactor thay vì che đi.

Khi nào NÊN và KHÔNG NÊN dùng interface?

Nên dùng interface:

  • Tầng Repository — có thể đổi database (MySQL → Spanner → MongoDB)
  • Tầng Service — có thể có nhiều implementation (Dynamic, Static, Custom)
  • Khi cần mock để viết unit test — interface giúp tạo mock dễ dàng với Mockito
  • Khi cần runtime switching (Strategy Pattern) — chọn implementation dựa trên condition

Không cần interface:

  • Class chỉ có duy nhất 1 implementation và không cần mock (ví dụ: utility class nội bộ)
  • Class là final hoặc không có ý định extend
  • Các @Configuration class — chúng không cần interface
⚠️ Anti-pattern phổ biến

Đừng tạo IUserService + UserServiceImpl khi chỉ có 1 implementation và không có kế hoạch thêm implementation nữa. Thêm code mà không thêm giá trị — đây là over-engineering kinh điển.

Unit Test với Mockito — Bằng chứng sống của Loose Coupling
Dành cho: Developer muốn thấy lợi ích thực tế của loose coupling qua code test.

Đây là phần thuyết phục nhất khi giới thiệu loose coupling cho người mới: viết unit test.

Thêm dependency vào pom.xml:

XML 
<!-- JUnit 5 + Mockito — Spring Boot Test starter đã bao gồm cả hai -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
    <!-- Bao gồm: JUnit 5, Mockito, AssertJ, Spring Test -->
</dependency>

Với tight coupling — gần như bất khả thi:

Java — ❌ Tight coupling test 
public class UserService {
    // ❌ Hard-coded dependency → không thể test mà không cần MySQL thật
    private final MySQLUserRepository repo = new MySQLUserRepository();
}

// Test phải có MySQL running → không phải unit test nữa, là integration test
@Test
void getById_shouldReturnUser() {
    UserService service = new UserService();
    // service.repo đang trỏ đến MySQL thật → test chạy chậm, fragile
}

Với loose coupling — đơn giản và nhanh:

Java — ✅ UserService with constructor injection 
@Service
public class UserService {
    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public String getById(int id) {
        if (id <= 0) {
            throw new IllegalArgumentException("ID phải lớn hơn 0");
        }
        return userRepository.getById(id);
    }
}
Java — ✅ Unit Test với Mockito 
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

import static org.assertj.core.api.Assertions.assertThat;
import static org.assertj.core.api.Assertions.assertThatThrownBy;
import static org.mockito.ArgumentMatchers.anyInt;
import static org.mockito.Mockito.*;

// ✅ Unit test với Mockito — không cần database, không cần Spring Context
@ExtendWith(MockitoExtension.class)
class UserServiceTest {

    @Mock
    private UserRepository userRepository; // Mockito tạo implementation giả tại runtime

    @InjectMocks
    private UserService userService; // Mockito tự inject mock vào constructor

    @Test
    @DisplayName("getById() trả về đúng user khi ID hợp lệ")
    void getById_withValidId_returnsUser() {
        // Arrange
        int userId = 1;
        String expectedUser = "Nguyen Van A";
        when(userRepository.getById(userId)).thenReturn(expectedUser);

        // Act
        String actualUser = userService.getById(userId);

        // Assert
        assertThat(actualUser).isEqualTo(expectedUser);
        verify(userRepository, times(1)).getById(userId);
    }

    @Test
    @DisplayName("getById() throw exception khi ID không hợp lệ")
    void getById_withInvalidId_throwsException() {
        assertThatThrownBy(() -> userService.getById(-1))
                .isInstanceOf(IllegalArgumentException.class)
                .hasMessage("ID phải lớn hơn 0");

        verifyNoInteractions(userRepository); // Repository không được gọi
    }

    @Test
    @DisplayName("getById() propagate exception từ repository")
    void getById_whenRepositoryFails_throwsException() {
        when(userRepository.getById(anyInt()))
                .thenThrow(new RuntimeException("Database connection failed"));

        assertThatThrownBy(() -> userService.getById(1))
                .isInstanceOf(RuntimeException.class)
                .hasMessage("Database connection failed");
    }
}

Test chạy trong milli-seconds, không cần database, không cần Spring Context. Đây chính là "bằng chứng sống" của loose coupling — khi bạn viết test dễ, nghĩa là code đang có thiết kế tốt.

✅ Nguyên tắc vàng

Nếu viết unit test khó — đó là mùi của tight coupling. Hãy refactor thay vì bỏ qua test.

Key Takeaways

  • Tight Coupling là nguồn gốc của hệ thống khó maintain — fix 1 bug sinh ra 10 bug
  • IoC đảo ngược quyền điều khiển — class không tự tạo dependency nữa
  • DIP yêu cầu phụ thuộc vào abstraction (interface) thay vì concrete class
  • DI inject dependency từ bên ngoài qua constructor — class không biết và không cần biết ai cung cấp
  • IoC Container (Spring ApplicationContext) tự động hóa toàn bộ quá trình tạo và inject
  • Constructor injection là best practice — dùng được final, dễ test, dependency explicit
  • Loose coupling ≠ không phụ thuộc — các class vẫn phụ thuộc nhau, nhưng qua interface nên ít bị ảnh hưởng khi thay đổi
  • Circular dependency = dấu hiệu thiết kế có vấn đề — refactor thay vì dùng @Lazy để che đi
  • Unit test dễ viết = bằng chứng tốt nhất rằng code đang loose coupling

Bước tiếp theo

  • Unit test với Mockito — thực hành viết test cho từng service; bạn sẽ thấy lợi ích loose coupling ngay lập tức
  • Clean Architecture — kiến trúc xây dựng hoàn toàn trên nền tảng loose coupling, với các layer rõ ràng
  • SOLID Principles toàn bộ — DIP chỉ là 1 trong 5 nguyên lý; SRP và ISP đặc biệt hữu ích khi thiết kế interface
  • sealed interface (Java 17+) — cho phép kiểm soát chặt chẽ tập hợp implementation được phép tồn tại
  • Spring AOP — công cụ mạnh mẽ xây dựng trên nền IoC Container, dùng cho cross-cutting concerns (logging, transaction, security)
Câu hỏi tự kiểm tra

Sau khi đọc bài này, bạn nên trả lời được những câu hỏi sau mà không cần nhìn lại:

  1. Sự khác biệt giữa IoC (design principle) và DI (design pattern) là gì? IoC có thể tồn tại mà không có DI không?
  2. Tại sao constructor injection được ưu tiên hơn field injection (@Autowired)? Liệt kê ít nhất 3 lý do cụ thể.
  3. Nếu Spring throw NoUniqueBeanDefinitionException, nghĩa là gì và có bao nhiêu cách giải quyết?
  4. Circular dependency xảy ra khi nào? Cách tốt nhất để xử lý là gì?
  5. Bạn có thể giải thích tại sao singleton bean không được có mutable state không?
  6. Khi nào nên dùng @Qualifier và khi nào nên dùng @Primary?
  7. Tại sao unit test dễ viết hơn khi code dùng loose coupling? Mockito mock được interface hay concrete class tốt hơn?
  8. Bạn có thể vẽ dependency graph của một Spring Boot application đơn giản (Controller → Service → Repository) và giải thích Spring tạo các bean theo thứ tự nào không?
