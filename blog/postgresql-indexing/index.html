1. Tại sao cần hiểu sâu về Index?

Bạn đã bao giờ gặp tình huống này chưa?

Query chạy tốt trong môi trường dev với 10,000 dòng dữ liệu. Lên production với 50 triệu dòng — timeout sau 30 giây. Team hỏi: "Tại sao không đánh index từ đầu?"

Hoặc ngược lại:

Database có quá nhiều index — mỗi lần INSERT/UPDATE mất gấp đôi thời gian so với cần thiết. DBA chỉ ra rằng 40% các index chưa bao giờ được sử dụng.

Cả hai tình huống đều xuất phát từ cùng một vấn đề: hiểu không đủ sâu về cơ chế hoạt động của index. Đánh index không phải lúc nào cũng là giải pháp — và không đánh cũng không phải lúc nào cũng là lỗi. Quyết định đúng đòi hỏi phải hiểu:

  • PostgreSQL Planner quyết định dùng index hay không dựa trên điều gì?
  • Mỗi loại index hoạt động theo cơ chế nào và phù hợp với data pattern gì?
  • Trade-offs giữa read performance và write overhead khi thêm index là gì?

Bài viết này sẽ trả lời toàn bộ những câu hỏi đó — từng bước, có ví dụ SQL thực tế và visual để bạn dễ hình dung.

2. EXPLAIN và EXPLAIN ANALYZE — Đọc Query Plan

Trước khi nói đến index, bạn cần biết công cụ để đánh giá hiệu quả của index: lệnh EXPLAIN và EXPLAIN ANALYZE.

2.1 EXPLAIN — Query Plan mà không thực thi

EXPLAIN hiển thị query plan mà PostgreSQL Planner đã chọn — dựa trên thống kê bảng (pg_statistic) mà không thực thi thực sự.

SQL 
EXPLAIN SELECT * FROM engineer WHERE country_id = 10;
Seq Scan on engineer (cost=0.00..4821.00 rows=3200 width=48) -- cost=startup_cost..total_cost -- rows = ước tính số row trả về -- width = kích thước trung bình mỗi row (bytes) Filter: (country_id = 10)
FieldÝ nghĩaGhi chú
startup_costChi phí trả về row đầu tiênHash Join có startup cao vì phải build hash table trước
total_costChi phí ước tính toàn bộ kết quảPlanner tối ưu total_cost trừ khi có LIMIT
rowsSố row Planner ước tính sẽ trả vềLệch xa thực tế → chạy ANALYZE table_name
widthĐộ rộng trung bình (bytes) mỗi output rowSELECT gender (smallint) → width = 2

2.2 EXPLAIN ANALYZE — Thực thi và so sánh

EXPLAIN ANALYZE thực sự chạy query và so sánh ước tính với thực tế.

SQL 
EXPLAIN ANALYZE SELECT * FROM engineer WHERE country_id = 10;
Seq Scan on engineer (cost=0.00..4821.00 rows=3200 width=48) (actual time=0.043..18.234 rows=3187 loops=1) Filter: (country_id = 10) Rows Removed by Filter: 96813 Planning Time: 0.312 ms Execution Time: 18.891 ms
Field bổ sungÝ nghĩa
actual time = X..Y msX = startup thực tế, Y = hoàn thành thực tế. So sánh với cost để đánh giá Planner.
rows (actual)Số row thực sự trả về. Chênh lệch lớn với estimate → statistics lỗi thời.
loopsSố lần node được thực thi. actual time là trung bình per-loop. Tổng = actual_time × loops.

⚠️ Lưu ý quan trọng

Không dùng EXPLAIN ANALYZE với DML (INSERT/UPDATE/DELETE) vì nó thực thi thật. Nếu cần test:

SQL 
BEGIN;
EXPLAIN ANALYZE UPDATE engineer SET salary = 1000 WHERE id = 1;
ROLLBACK; -- Đảm bảo không commit

2.3 EXPLAIN Options hữu ích

SQL 
-- Xem cache hits vs disk reads
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM engineer WHERE country_id = 10;

-- Output JSON — dùng với explain.depesz.com hoặc pev2
EXPLAIN (ANALYZE, FORMAT JSON) SELECT * FROM engineer WHERE country_id = 10;

-- Tắt per-row timer để giảm overhead đo thời gian
EXPLAIN (ANALYZE, TIMING OFF) SELECT * FROM engineer WHERE country_id = 10;

3. Sequential Scan — Baseline hiểu index

Sequential Scan (Seq Scan) là phương thức quét mặc định: đọc tuần tự toàn bộ heap (bảng chính) từ đầu đến cuối — không cần index.

Cơ chế

Duyệt qua từng row theo thứ tự vật lý trên disk. Data lưu dạng pages (8KB/page). PostgreSQL đọc nguyên page dù chỉ cần 1 row.

Khi Planner chọn Seq Scan

(1) Không có index phù hợp. (2) Query trả về > ~5–15% tổng số row. Với tỷ lệ cao, Seq Scan nhanh hơn index scan do tránh random I/O.

Vấn đề

Data có thể bị phân mảnh (fragmentation) → nhiều I/O hơn. Bảng 50M row, mỗi query đều Seq Scan → nghẽn cổ chai nghiêm trọng.

💡 Seq Scan không phải lúc nào cũng xấu

Planner thông minh hơn bạn nghĩ. Nếu query trả về 30% bảng, Index Scan sẽ gây nhiều random I/O hơn Seq Scan. Đừng force index nếu Planner chọn Seq Scan — hãy đọc EXPLAIN ANALYZE trước.

🔍 Cơ chế lookup

Index lưu: (indexed_value, TID) → biết value cần tìm → biết ngay row ở đâu trên disk. Như mục lục cuốn sách: thay vì đọc từ trang 1, tra mục lục → đến đúng trang.

Ba chiến lược quét chính

Index Scan

Tra index → lấy TID → fetch row từ heap. Tốt khi số row kết quả nhỏ. Có thể gây nhiều random I/O nếu row phân tán nhiều page.

Bitmap Index Scan

Tập hợp nhiều TID → sort → fetch heap theo thứ tự vật lý. Giảm random I/O. Planner chọn khi số row trung bình hoặc có nhiều điều kiện OR.

Index-Only Scan

Đọc hoàn toàn từ index, không truy cập heap. Cực nhanh khi Visibility Map xác nhận page all-visible. Cần Covering Index (INCLUDE).

Trade-offs khi tạo index

Khi nào KHÔNG nên tạo index

  • Bảng rất nhỏ (< vài nghìn row) — Seq Scan thường nhanh hơn
  • Column cardinality cực thấp (boolean, gender) và không kết hợp với column khác
  • Bảng có tỉ lệ write >> read (log tables, audit tables)
  • Column hiếm khi xuất hiện trong WHERE, JOIN, ORDER BY

Các trường hợp B-Tree hoạt động tốt

SQL 
-- Equality
WHERE email = 'user@example.com'

-- Range queries
WHERE salary BETWEEN 5000 AND 10000
WHERE created_at >= '2024-01-01'

-- Prefix match (ONLY prefix — không phải suffix!)
WHERE name LIKE 'abc%'   -- ✅ dùng được index
WHERE name LIKE '%abc'   -- ❌ KHÔNG dùng được B-Tree (cần GIN + pg_trgm)

-- ORDER BY hiệu quả (không cần extra sort step)
CREATE INDEX idx_dept_salary ON employee (dept_id ASC, salary DESC);
SELECT * FROM employee ORDER BY dept_id ASC, salary DESC LIMIT 10;

Composite Index — Chiến lược

SQL — Tạo index production-safe 
-- CONCURRENTLY: không lock table trong quá trình tạo index
-- Bắt buộc trên production table đang có traffic
CREATE INDEX CONCURRENTLY idx_orders_user_status
    ON orders (user_id, status);

-- Với Composite (x, y):
-- Query chỉ x       → dùng được composite (x, y)
-- Query chỉ y       → cần index riêng trên (y)
-- Query cả x và y   → composite (x, y) hiệu quả hơn 2 index riêng

Hash Index
  • Storage nhỏ hơn (chỉ lưu hash, không lưu actual value)
  • Equality =: O(1) avg — nhanh hơn B-Tree
  • ❌ Không hỗ trợ range (<, >, BETWEEN)
  • ❌ Không hỗ trợ ORDER BY
  • ❌ Không thể tạo Composite index
  • WAL-safe từ PostgreSQL 10+
B-Tree (so sánh)
  • Storage lớn hơn (lưu actual value)
  • Equality =: O(log N) — chậm hơn chút với key rất dài
  • ✅ Hỗ trợ range, ORDER BY, LIKE 'prefix%'
  • ✅ Hỗ trợ Composite index
  • ✅ Mọi phiên bản PostgreSQL
  • Default — dùng cho hầu hết trường hợp
SQL — Khi nào dùng Hash Index 
-- Chỉ dùng khi: pattern chỉ có equality (=) VÀ column có giá trị dài
-- UUID, session token, access token, long API key...
CREATE INDEX idx_session_token ON sessions USING HASH (session_token);

-- Query phải là equality thuần túy
SELECT * FROM sessions WHERE session_token = 'abc123xyz-long-token...';
⚠️ Thực tế production

Trong hầu hết trường hợp, B-Tree vẫn đủ nhanh cho equality và linh hoạt hơn nhiều. Chỉ chuyển sang Hash khi đã profiling và xác nhận bottleneck thật sự tại equality lookup trên column giá trị rất dài.

7. GiST và SP-GiST — Geometry, Full-text, Ranges

7.1 GiST Index

Extensible
Generalized Search Tree — framework mở rộng được

GiST phù hợp cho dữ liệu phức tạp không có thứ tự tuyến tính: geometry, full-text, ranges, network. Hỗ trợ các operators đặc biệt:

OperatorÝ nghĩa
@>contains (bao gồm)
<@is contained by
&&overlaps (giao nhau)
<->distance — KNN search
<< / >>strictly left/right of
SQL — GiST cho PostGIS và Full-text 
-- KNN search: tìm 10 địa điểm gần nhất với tọa độ (106.66, 10.76) — TP.HCM
CREATE INDEX idx_locations_geom ON locations USING GIST (geom);

SELECT name, geom <-> ST_MakePoint(106.66, 10.76) AS distance
FROM locations
ORDER BY geom <-> ST_MakePoint(106.66, 10.76)
LIMIT 10;

-- GiST cho full-text search (GIN thường tốt hơn cho FTS — xem mục 8)
CREATE INDEX idx_docs_tsv ON documents USING GIST (to_tsvector('english', content));

SELECT * FROM documents
WHERE to_tsvector('english', content) @@ to_tsquery('postgres & index');

7.2 SP-GiST Index

Space-Partitioned
Quadtrees, k-d trees, radix trees

SP-GiST hỗ trợ cấu trúc dữ liệu phân vùng không gian. Phù hợp dữ liệu phân bố không đều hoặc phân cụm.

-- SP-GiST tốt hơn GiST cho network lookups
CREATE INDEX idx_sessions_ip ON sessions USING SPGIST (client_ip);

SELECT * FROM sessions WHERE client_ip <<= '192.168.1.0/24';
-- <<= : is contained within or equal (subnet match)

8. GIN Index — JSONB, Arrays, Full-text

Inverted Index
Generalized Inverted Index — ánh xạ element → rows

GIN là inverted index — ánh xạ từng element value đến danh sách row chứa nó. Tương tự inverted index trong search engine. Tốt nhất cho JSONB, arrays, full-text search.

OperatorÝ nghĩaData type
@>containsarray, jsonb, hstore
<@contained byarray, jsonb
&&overlaparray
?key existsjsonb, hstore
?|any key existsjsonb, hstore
?&all keys existjsonb, hstore
@@text search matchtsvector
SQL — GIN cho JSONB, Array, và LIKE '%keyword%' 
-- GIN trên JSONB — tìm sản phẩm có attribute color = "red"
CREATE INDEX idx_products_attrs ON products USING GIN (attributes);
SELECT * FROM products WHERE attributes @> '{"color": "red"}';

-- GIN trên array — bài viết có cả 2 tag
CREATE INDEX idx_articles_tags ON articles USING GIN (tags);
SELECT * FROM articles WHERE tags @> ARRAY['postgresql', 'index'];

-- GIN + pg_trgm: LIKE '%keyword%' — substring search
-- (B-Tree chỉ hỗ trợ prefix LIKE 'prefix%')
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_products_name_trgm ON products USING GIN (name gin_trgm_ops);
SELECT * FROM products WHERE name ILIKE '%wireless%';

-- GIN cho full-text search — tốt hơn GiST với FTS
CREATE INDEX idx_docs_fts ON documents USING GIN (to_tsvector('english', content));
SELECT * FROM documents
WHERE to_tsvector('english', content) @@ to_tsquery('postgres & performance');
⚠️ GIN Write Overhead

GIN update tốn kém hơn B-Tree (cập nhật nhiều posting lists). PostgreSQL mặc định bật fastupdate=on để batch writes — nhưng query sau đó có thể cần flush pending list. Dùng fastupdate=off nếu cần read latency nhất quán.

9. BRIN Index — Bảng khổng lồ, Insert-ordered

Ultra-compact
Block Range INdex — summary per block range

BRIN lưu summary (min/max) cho từng block range (128 pages mặc định). Kích thước cực nhỏ, phù hợp bảng lớn có dữ liệu correlated với thứ tự vật lý.

Index size
~vài KB
Dù bảng có hàng tỷ row — vì chỉ lưu min/max per 128 pages
Điều kiện áp dụng
Correlated
Dữ liệu phải physically correlated với thứ tự insert (timestamp tăng dần)
Không phù hợp
UUID, Random
UUID random hoàn toàn — min/max không loại được nhiều block
SQL — BRIN cho time-series logs 
-- pages_per_range=64: chính xác hơn (mặc định 128)
-- Giảm pages_per_range → chính xác hơn, nhưng index lớn hơn
CREATE INDEX idx_logs_created BRIN ON logs (created_at)
    WITH (pages_per_range = 64);

-- Hiệu quả vì created_at tăng dần theo insert order
-- → Planner có thể bỏ qua hầu hết blocks không thỏa điều kiện
SELECT * FROM logs
WHERE created_at >= '2024-01-01'
  AND created_at <  '2024-02-01';
✅ Use case điển hình

Bảng events, logs, transactions, sensor_readings — hàng tỷ row, insert theo thứ tự thời gian, query theo range timestamp. BRIN ở đây nhỏ hơn B-Tree hàng trăm lần và đủ hiệu quả để loại bỏ hầu hết blocks không cần thiết.

10. Partial, Covering và Expression Index

10.1 Partial Index — Nhỏ hơn, nhanh hơn

Pattern
Index chỉ trên tập con rows thỏa điều kiện WHERE
SQL — 3 pattern Partial Index phổ biến nhất 
-- Pattern 1: Chỉ index active records
-- Index nhỏ hơn vì chỉ cover ~5% bảng (orders đang pending)
CREATE INDEX idx_orders_pending_created
    ON orders (created_at)
    WHERE status = 'pending';

-- Query MÀ bạn thường dùng
SELECT * FROM orders
WHERE status = 'pending'
  AND created_at > NOW() - INTERVAL '7 days';

-- Pattern 2: Loại trừ NULL values — index nhỏ hơn, lookup sạch hơn
CREATE INDEX idx_users_phone
    ON users (phone_number)
    WHERE phone_number IS NOT NULL;

-- Pattern 3: Unique constraint trên subset
-- Email unique chỉ với active (non-deleted) users
CREATE UNIQUE INDEX idx_users_active_email
    ON users (email)
    WHERE is_deleted = FALSE;
✅ Tại sao Partial Index mạnh?

Partial Index thường nhỏ hơn full index 5–20x → faster lookup, ít RAM hơn trong shared_buffers, write overhead thấp hơn (chỉ update index khi row thỏa điều kiện WHERE của index).

10.2 Covering Index — Index-Only Scan

PostgreSQL 11+
Mệnh đề INCLUDE — đọc từ index, không cần heap
SQL — Covering Index với INCLUDE 
-- user_id là key column (dùng để tìm kiếm)
-- email, full_name là included columns (chỉ để đọc, không dùng để sort/filter)
CREATE INDEX idx_users_covering
    ON users (user_id)
    INCLUDE (email, full_name);

-- Query này → Index-Only Scan (không fetch heap!)
SELECT email, full_name
FROM users
WHERE user_id = 42;

-- Lưu ý: VACUUM thường xuyên cập nhật Visibility Map
-- → tăng tỉ lệ Index-Only Scan thành công
VACUUM ANALYZE users;
💡 Index-Only Scan vs Index Scan

Index-Only Scan đọc data trực tiếp từ index leaf nodes, bỏ qua heap hoàn toàn. Điều kiện: (1) tất cả columns cần SELECT đều có trong index, (2) Visibility Map xác nhận page all-visible (sau VACUUM). Đây là query path nhanh nhất có thể.

10.3 Expression Index (Functional Index)

SQL — Index trên biểu thức/function 
-- Case-insensitive email search
-- Query PHẢI dùng cùng biểu thức: lower(email)
CREATE INDEX idx_users_email_lower ON users (lower(email));
SELECT * FROM users WHERE lower(email) = 'user@example.com';

-- Index trên JSON field extraction
CREATE INDEX idx_events_type ON events ((payload->>'event_type'));
SELECT * FROM events WHERE payload->>'event_type' = 'purchase';

-- Index trên computed value — năm của created_at
CREATE INDEX idx_orders_year ON orders (EXTRACT(YEAR FROM created_at));
SELECT * FROM orders WHERE EXTRACT(YEAR FROM created_at) = 2024;
⚠️ Function phải IMMUTABLE

Chỉ các function IMMUTABLE mới được phép trong Expression Index. NOW(), RANDOM(), CURRENT_TIMESTAMP — không được phép vì giá trị thay đổi theo thời gian.

11. Giám sát và Bảo trì Index

Phát hiện index không được dùng

SQL — pg_stat_user_indexes 
-- Tìm index CHƯA BAO GIỜ được dùng — candidates for DROP
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan AS times_used
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelid NOT IN (
      -- Bỏ qua PRIMARY KEY và UNIQUE constraints
      SELECT indexrelid FROM pg_index WHERE indisprimary OR indisunique
  )
ORDER BY pg_relation_size(indexrelid) DESC;

-- Thống kê đầy đủ cho một bảng cụ thể
SELECT
    indexname,
    idx_scan       AS scans,
    idx_tup_read   AS tuples_read,
    idx_tup_fetch  AS tuples_fetched,
    pg_size_pretty(pg_relation_size(indexrelid)) AS size
FROM pg_stat_user_indexes
WHERE tablename = 'orders'
ORDER BY idx_scan DESC;
⚠️ Lưu ý khi DROP index

Stats được reset sau mỗi lần restart PostgreSQL. Đừng DROP index chỉ vì idx_scan = 0 — đảm bảo server đã chạy ít nhất vài tuần và stats là accurate. Dùng DROP INDEX CONCURRENTLY trên production để không lock table.

Xử lý Index Bloat

SQL — Maintenance commands 
-- Rebuild index không lock table (PostgreSQL 12+)
REINDEX INDEX CONCURRENTLY idx_engineer_country;

-- VACUUM: cập nhật Visibility Map + thu hồi dead tuples
-- → tăng tỉ lệ Index-Only Scan thành công
VACUUM ANALYZE engineer;

-- Cập nhật statistics sau bulk insert/update
ANALYZE engineer;

-- Tăng statistics target cho column có distribution phức tạp
-- (mặc định = 100; range 1-10000)
ALTER TABLE engineer
    ALTER COLUMN country_id SET STATISTICS 500;
Deep Dive
MVCC và Index Bloat — Tại sao index phình to?
Dành cho: DBA và developer muốn hiểu tại sao cần VACUUM định kỳ và HOT update giúp gì.

PostgreSQL dùng MVCC (Multi-Version Concurrency Control) — thay vì update in-place, mỗi UPDATE tạo một version mới của row (tuple) và đánh dấu version cũ là "dead".

Hậu quả với index: mỗi version của row đều có một entry trong index. Sau nhiều UPDATE, index chứa hàng loạt pointer tới dead tuples → index bloat.

HOT (Heap Only Tuple) là optimization: nếu UPDATE không thay đổi cột index nào, PostgreSQL tạo version mới trong cùng page với old tuple và không update index — tiết kiệm đáng kể write overhead.

SQL — Kiểm tra index bloat 
-- Tỉ lệ sử dụng thực tế của index (sau bloat)
SELECT
    indexrelid::regclass AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS total_size,
    round(100 * pg_stat_get_dead_tuples(relid)::numeric
          / NULLIF(pg_stat_get_live_tuples(relid) + pg_stat_get_dead_tuples(relid), 0), 2)
        AS dead_tuple_pct
FROM pg_index
JOIN pg_stat_user_tables ON relid = pg_stat_user_tables.relid
WHERE pg_relation_size(indexrelid) > 10 * 1024 * 1024 -- index > 10MB
ORDER BY dead_tuple_pct DESC NULLS LAST;
✅ Chiến lược VACUUM

Bật autovacuum (mặc định ON) và tune autovacuum_vacuum_scale_factor cho bảng write-heavy. Với bảng rất lớn, cân nhắc vacuum_cost_delay để giảm impact lên production. Chạy VACUUM ANALYZE thủ công sau bulk operation lớn.

Quick Decision — Tình huống → Giải pháp

Tình huống Giải pháp
General WHERE / ORDER BY / JOIN B-Tree (default — không cần USING)
Equality trên UUID / session token / long string Hash Index
Chỉ query active rows / non-null / subset Partial Index (WHERE clause)
SELECT chỉ cần vài column, tránh fetch heap Covering Index (INCLUDE)
WHERE lower(col) = ... / biểu thức function Expression Index
JSONB / arrays: @>, <@, &&, ? GIN Index
Full-text search (@@ to_tsquery) GIN (read-heavy) · GiST (write-heavy)
Geometry, KNN, PostGIS, ranges GiST Index

Timestamp bảng hàng tỷ row, insert-ordered BRIN Index
LIKE '%keyword%' (substring, không phải prefix) GIN + pg_trgm extension

BRIN

13. Kết luận — Framework để quyết định

Indexing không phải là "thêm index = nhanh hơn". Quyết định đúng đòi hỏi framework tư duy:

  1. Đo trước, tối ưu sau — luôn bắt đầu bằng EXPLAIN ANALYZE. Đừng đoán.
  2. Hiểu data pattern — cardinality, distribution, query pattern, tỉ lệ read/write.
  3. Chọn index type phù hợp — B-Tree mặc định, chuyển sang specialized type khi có use case rõ ràng.
  4. Đo lại sau khi tạo index — đảm bảo Planner thực sự dùng index và performance cải thiện.
  5. Giám sát định kỳ — pg_stat_user_indexes, index bloat, VACUUM schedule.
  6. Xóa index không dùng — index thừa làm chậm write mà không có lợi gì.

Key Takeaways

  • B-Tree là default — hiệu quả cho hầu hết trường hợp: equality, range, ORDER BY, JOIN
  • EXPLAIN ANALYZE là công cụ số một — đọc được query plan = biết cần làm gì
  • Partial Index thường nhỏ hơn full index 10x và nhanh hơn đáng kể cho workload phân bố lệch
  • Covering Index (INCLUDE) cho phép Index-Only Scan — query path nhanh nhất
  • GIN là lựa chọn tốt nhất cho JSONB, arrays, và full-text search
  • BRIN cực kỳ nhỏ gọn cho bảng time-series hàng tỷ row
  • MVCC tạo ra dead tuples theo thời gian — VACUUM ANALYZE là bảo trì bắt buộc
  • CREATE INDEX CONCURRENTLY — luôn dùng trên production để tránh lock table
Câu hỏi tự kiểm tra

Sau khi đọc bài này, thử trả lời những câu hỏi sau:

  1. Sự khác biệt giữa Index Scan, Bitmap Index Scan, và Index-Only Scan là gì? Khi nào Planner chọn từng loại?
  2. Tại sao WHERE name LIKE '%abc' không dùng được B-Tree index? Phải làm gì để optimize query này?
  3. Composite index (a, b) — query WHERE b = ? có dùng được không? Tại sao?
  4. Khi nào BRIN index hiệu quả và khi nào hoàn toàn vô dụng? Cho ví dụ cụ thể.
  5. Điều kiện để Index-Only Scan xảy ra là gì? VACUUM có vai trò gì trong đây?
  6. Partial Index WHERE status = 'pending' — query WHERE status = 'completed' có dùng được index này không? Tại sao?
  7. Tại sao GIN update tốn kém hơn B-Tree? fastupdate=on giải quyết vấn đề gì?
  8. Index bloat xảy ra do nguyên nhân gì? Cách phát hiện và xử lý?

Trả lời được 6/8 câu → bạn nắm vững nội dung. Dưới 6 → đọc lại phần còn thắc mắc và thực hành với EXPLAIN ANALYZE trên database thực tế.
